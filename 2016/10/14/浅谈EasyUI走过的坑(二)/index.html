闲聊EasyUI以及项目中踩过的坑(二)：EasyUI Combotree 组合树中常见问题

经验分享
  • EasyUI
  • jQuery

上篇文章简单介绍了EasyUI，以及环境的搭建，这篇文章着重分享自己在项目中遇到的Combotree即组合树控件使用问题


了解Combotree

  • 扩展自 $.fn.combo.defaults 和 $.fn.tree.defaults。
  • 通过$.fn.combotree.defaults 重写默认的 defaults。
  • 组合树（combotree）把选择控件和下拉树结合起来。它与组合框
  • （combobox）相似，不同的是把列表替换成树组件。组合树
  • （combotree）支持带有用于多选的树状态复选框的树。

下图为包含多选框的Combotree

上面是引用了jQuery-EasyUI中文网的Combotree简介。简单说就是Combo(组合)的扩展，与（combobox）很类似，但用法上比（combobox）复杂，实际项目中调试组合树也相对耗时间。

其中事件，方法的详细说明就不在这里列举，放上Combotree官方文档，可以 更详细的认识。

项目中combotree遇到的问题

由于项目中通常是多个控件一块使用，比较常见的组合有（combotree）组合树，（combobox）组合框，（Datagrid）数据网格，（Linkbutton）链接按钮等等，这就容易造成最近项目中遇到的如下问题。

相关联的控件加载速度不同，引起的问题

下面分别是Datagrid数据网格，combotree组合树的标签创建格式

    data-options="url:'get_data.php',required:true">

combotree组合树标记创建格式

可以看到两个控件的数据来源都是data-options后跟上url来获取的json数据，可以是本地的数据源，也可以是后台传过来的json串，那么在默认情况下进入页面时，两个控件都会请求各自的url来加载，如果两个控件之间没有关联性，可能不太会影响显示结果，但如果两个或多个控件之间存在关联，就会造成些未知的问题。

举个栗子，就拿公司最近的项目来说吧有个页面的逻辑为所属机构（combotree）下关联所属台区（combobox），以及一个含下级（checkbox）标志来决定所属终端（Datagrid）的显示，其中与所属终端有直接关联的只有所属台区，而台区与所属机构，含下级有关联。

当选中所属机构，以及是否选中含下级，才能决定所属台区的内容，而所属终端的查询条件是根据所属台区走的，这样就存在一个隐患:全部控件在页面加载时同时请求，但当树的内容过多，还没能加载完毕，数据表格的请求已经在后台开始处理了。后续出的各种处理肯定是不对的。

后来的解决方案是，当树的内容加载完毕（这里的树内容不多，采用全部加载，如果树的分支很多就要采用异步加载），Ajax请求加载相对应的台区（通常checkbox的响应速度是很快的，当树加载完毕，它肯定加载完成了），监听台区加载完成后，再去获取台区内容，加载所属终端，这样一来由于加载速度引起的问题也就迎刃而解了。

具体的处理逻辑见下面的代码块：

<td class="td-title">所属机构：</td>
<td class="td-content"><input id="f_tID"
    class="easyui-combotree" style="width:150px;height:26px;"
    data-options="url:'../treeJ_queryAgencyTreeForUser',
    method:'post',
    editable:false,
    onLoadSuccess:function() {
        var f_tID = $('#hidF_tID').val();
        var tree = $('#f_tID').combotree('tree');
        if(f_tID == ''){
            f_tID = tree.tree('getRoot').id;
        }
        var node = tree.tree('find', f_tID);
        $('#f_tID').combotree('setValue', f_tID);
        $('#datainfo').datagrid({url:'terminalJ_getTerminalMngInfo'});
    },
    onChange:function(newValue,oldValue){
       $('#f_tID').combotree('setValue',newValue);
       if(newValue!=oldValue){
           $('#f_transformerID').combobox({
            url:'terminalJ_getTransformer',valueField:'id',
            textField:'f_transformerName',
            method:'post',
            valueField:'id',
            panelHeight:'auto',
            editable:false,
            queryParams:{
                underFlg: $('#underFlg').is(':checked'),
                f_tID: newValue
               },

                var f_transformerID = $('#hidF_transformerID').val();
                if(f_transformerID != '') {
                    $('#f_transformerID').combobox('setValue',f_transformerID)
                }
            }
           });
               }else{
                return;                                               
               }
        }
">

具体说明如下：

onLoadSuccess为Tree的事件，combotree也同样继承了，跟字面意思一样，当树的内容加载完成后会触发该方法。

这里的处理为，当机构树加载完成后，再给数据表格url，数据表格才会去后台请求数据（这里页面第一次访问，选中的树为根节点，即所属台区为全部台区，后台也是同样处理查询全部的请求）。

onChange为Combo的事件，combotree也同样继承了，当树所选的内容改变后会触发该方法。

这里的处理为监听所选树的内容，当内容确实是改变后，根据含下级标志，以及所选机构ID异步请求台区的内容，并且是台区组合框加载完毕后，根据隐藏域的值，选中台区。

含下级单选框的监听放到了如下的js代码块中：

//监听含下级选中框，每改变一次，重新请求刷新所属台区的内容
  $("#underFlg").change(function() {
    $('#f_transformerID').combobox({
        url : 'terminalJ_getTransformer',
        valueField : 'id',
        textField : 'f_transformerName',
        method : 'post',
        valueField : 'id',
        panelHeight : 'auto',
        editable : false,
        queryParams : {
            underFlg : $('#underFlg').is(':checked'),
            f_tID : $('#f_tID').combotree('getValue')
        }
    });
});

同样是一样的思路，监听含下级选中框，改变后会将当前的选中状态以及机构ID传到后台，根据传回的json异步刷新台区下拉框。

总结如下：

同类加载速度引起的问题，都是这种解决思路：让相关联的控件的第一环先加载完毕，再去加载下一环，以此类推。如果加载数据量过多，（例如combotree这一类）又对加载的速度有要求的情况下，可以使用异步加载，例如首次访问combotree只加载根层树节点，Datagrid进行分页只显示前50条数据等等。

树的数据结构都是采用一个数组中存放多个对象来实现，对象中可以使用children继续包括子对象，text即为对象的文本内容，state为其中的节点状态closed为关闭状态，open即为打开状态，iconCls为节点的图标，attributes为自定义的一些属性，id为远程数据需要的定位标志。

也同样可以使用url来加载服务器端传来的json数据

下图为项目中机构树数据库表部分结构，ID为自身id，F_FID为其直接的父id，F_PATH为含下级查询时提供方便，通过自己的F_PATH可清楚的获得自己所处的层级，以及自己上层的所有父节点。


用于测试的数据
  1. 在后台获取整棵树的结构，先可以取出根节点，即F_FID为空的节点id，然后依次取出第一层节点，编写回调函数即可全部取出所有的子节点，整个结构存储到集合中，再通过Struts2或者SpringMVC返回json类型的数据，传入到对应的url中。
  2. 在查询含下级的情况，先传到后台对应的id，以及含下级的标志，当选中含下级时，可编写类似下面的sql，及查询自己的下级所有节点信息，并加上自己的节点信息，这里假设对应的ID为297ed61e577568f7015775c4594b0003，查询的结果为3条记录，如下图：
SELECT
    ID,
    F_FID,
    F_NODECODE,
    F_PATH
FROM
    T_BUS_AGENCYTREE
WHERE
    F_PATH LIKE '%297ed61e577568f7015775c4594b0003%'
    OR ID = '297ed61e577568f7015775c4594b0003';

满足条件的查询结果

可以看出在树的结构表中添加了F_PATH这个字段，sql编写与操作整个树也相对容易了很多。

combotree的取值/写值问题

通常使用jQuery操作标签，使用$("#id").val()即可取出对象的值，这个取值命令在tree以及combotree中并不可用。

标签中加入如下内容，表明前后台传值为id，而前台显示是采用text。

data-options=
"
    url:'../treeJ_queryAgencyTreeForUser',
    method:'post',
    valueField:'id',
    textField:'text',
    editable:false
"

$('#id').combotree('getValue')这样的写法，取出树中的值id，显示即为对应的text的值，并且通常给后台传的也只有id才有意义。

$('#id').combotree('setValue'，XXX)这样可以将id的值写入，在存在该id的情况下，树节点会自动选中对应的该id的text的值。

对于如下数据，当setValue为11时，树会默认选中Java。

[{
    id: 1,
    text: 'Languages',
    children: [{
        id: 11,
        text: 'Java'
        },{
        id: 12,
        text: 'C++'
    }]
}]

又例如下面的操作：

onLoadSuccess:function() 
{

     var f_tID = '<s:property value='f_tID'/>';
     if(f_tID != '') {
         $('#f_tID').combotree('setValue',f_tID)
     } else {
            var tree = $('#f_tID').combotree('tree')
            $('#f_tID').combotree('setValue',tree.tree('getRoot').id)
        }
}

这里需要说明一点，对于查询条件，用户一般是喜欢自动保存的，例如我选中一个树节点进行修改的操作后，返回主页面我希望同时树还是选中当前的节点，方便我进行对比查看。

为了这种需求，我在后台将选中的树id存入了值域，到页面中借助Struts2的S标签：<s:property value='f_tID'/>进行取值，当值域有对应的值，我就给combotree附上取出的id，如果没有存值（一般为第一次进入页面）默认选中根节点。

选中根节点需要用到tree属性，$('#f_tID').combotree('tree').tree('getRoot').id即可获取当前树的根节点id，在树结构加载完毕后选中根节点的逻辑就完成了。

combotree中使用中容易出问题的地方，以及解决的思路在这里粗略的列举出来，当后续遇到了combotree其他的使用问题，会及时更新。
